#### Подключение к контейнеру
Запускаем контейнер, подключаемся к нему через sh, устанавливаем переменные окружения из файла .env, который монтируется при создании контейнера. Флаг --rm говорит о том, что после завершения работы контейнера
```sh
docker compose run --rm --entrypoint sh db-migrator -c "set -a && . /migrations/.env && set +a && sh"
```

#### Проверка наличия переменных
```sh
env             # Вывод всех переменных окружения
env | grep DB_  # Вывод переменных с префиксом DB_
```

#### Переход в директорию
Перейти в директорию с миграциями
```sh
cd migrations
```

#### Команды миграций для основной БД
```sh
# Проверить статус
./migrate-postgresql.sh main_db version

# Применить миграции  
./migrate-postgresql.sh main_db up

# Откатить миграции  
./migrate-postgresql.sh main_db down

# Создать миграцию
./migrate-postgresql.sh main_db create add_users_table
```

#### Команды миграций для основной БД
Выполняются те же команды, что и для основной БД, только вместо main_db
указывается analytic_db, например:
```sh
# Проверить статус
./migrate-postgresql.sh analytic_db version
```

#### Перенумерация миграций
Скрипт `renum.py` сдвигает нумерацию файлов миграций (удобно при вставке миграции в середину):
```sh
# Из корня проекта
python3 db_migrator/renum.py main_db 000016 up --step 2 --dry-run   # предпросмотр
python3 db_migrator/renum.py main_db 000016 up --step 2             # выполнить

# Не трогать файлы с нестандартными префиксами (например 100020)
python3 db_migrator/renum.py main_db 000016 up --upper 100000
```

#### Завершение работы
После выполнения необходимых команд, остановить работу контейнера, чтобы освободить ресурсы
```sh
exit # выход из консоли контейнера
docker-compose stop db_migrator # остановка контейнера
```
